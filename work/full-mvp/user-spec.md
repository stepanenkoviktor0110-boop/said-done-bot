# User Spec: said-done-bot — Full MVP

## Что делаем

Telegram-бот, который принимает голосовые сообщения, транскрибирует их и показывает два варианта действия: **📋 Извлечь задачи** или **📝 Сделать резюме**. Пользователь выбирает кнопку — бот выполняет выбранное действие.

При извлечении задач бот убирает словесный мусор, разрешает неопределённые ссылки ("тот мужик" → "уточнить у Ани"), упорядочивает задачи по зависимостям. После списка задач — фидбек-флоу с оценкой 1-5.

При резюме — краткое изложение от первого лица без фидбека.

Ведётся база пользователей с триал-системой (20 бесплатных → survey → +10 → блок). Несколько пересланных голосовых одновременно обрабатываются как единый контекст.

## Зачем

Предприниматели, менеджеры и люди с высокой когнитивной нагрузкой говорят как думают — хаотично, с оговорками. Сейчас нет инструмента, который превращает такую речь в план действий ИЛИ резюме — по выбору. ToTalk-ToDo решает эту проблему: переслал голосовое → выбрал что нужно → получил результат.

## Сценарии

### Сценарий 1: Одно голосовое — выбор действия

1. Пользователь пересылает голосовое сообщение боту
2. Бот: "🎙️ Распознаю голосовое..." (если >5 сек — статус)
3. Бот показывает расшифровку + две кнопки: [📋 Извлечь задачи] [📝 Сделать резюме]
4. Пользователь нажимает одну из кнопок

#### 1A: Пользователь нажал "📋 Извлечь задачи"
5. Бот: "🧠 Извлекаю задачи..."
6. LLM анализирует транскрипт, возвращает нумерованный список задач
7. Под списком — inline-кнопки оценки: 1 2 3 4 5
8. Пользователь нажимает оценку
9. Если оценка = 5 → бот благодарит, фидбек завершён
10. Если оценка < 5 → бот просит текстовый комментарий
11. После комментария → бот спрашивает: "Можно сохранить голосовое для улучшения качества? Да / Нет"
12. Если "Да" → аудио сохраняется. Если "Нет" → нет.
13. Бот благодарит

#### 1B: Пользователь нажал "📝 Сделать резюме"
5. Бот: "🧠 Анализирую голосовое..."
6. LLM делает краткое резюме от первого лица
7. Бот показывает резюме — фидбек НЕ запрашивается
8. Списание 1 кредита из триала

### Сценарий 2: Несколько голосовых одновременно

1. Пользователь пересылает 2-10 голосовых одновременно
2. Бот получает первое → запускает буфер 3 секунды
3. За время буфера приходят остальные голосовые
4. Бот транскрибирует все, объединяет в единый контекст
5. Показывает общую расшифровку + кнопки [📋 Извлечь задачи] [📝 Сделать резюме]
6. Далее — как в Сценарии 1A или 1B

**Edge cases:**
- Одно из голосовых не транскрибировалось → обрабатывает остальные, показывает "1 из 3 не удалось"
- Максимум 10 голосовых в пачке — больше → просит разбить
- Голосовое после закрытия буфера → новый отдельный запрос

### Сценарий 3: Первый контакт

1. Пользователь отправляет /start
2. Бот: приветствие + инструкция + "У тебя 20 бесплатных обработок"
3. Пользователь автоматически регистрируется в БД

### Сценарий 4: Триал исчерпан

1. Пользователь отправляет голосовое
2. Бот: "Лимит исчерпан. Ответь на 4 вопроса — получишь ещё 10"
3. Бот задаёт вопросы по одному:
   - "Что тебе понравилось в боте?"
   - "Что не понравилось или работало плохо?"
   - "Чего не хватает?"
   - "Общая оценка от 1 до 5"
4. Каждый ответ проверяется LLM на адекватность (sanity check)
5. Мусор (1-я попытка) → "ответь нормально"
6. Мусор повторно (2-я попытка) → блокировка навсегда
7. Все 4 ответа адекватные → +10 голосовых
8. После 30 (20+10) — полная блокировка

**Edge cases:**
- Бросил форму на середине → при следующем голосовом продолжает с того вопроса
- Получил отказ → повторная попытка недоступна
- LLM недоступен при sanity check → ответ принимается (fail-open)

### Сценарий 5: Не-голосовое сообщение

1. Пользователь отправляет текст, фото, стикер
2. Бот: "Я работаю только с голосовыми. Перешли мне голосовое сообщение"

### Сценарий 6: Прерывание фидбека новым голосовым

1. Пользователь получил задачи, но не нажал оценку
2. Переслал новое голосовое
3. Бот: предыдущий фидбек отброшен, обрабатывает новое голосовое

## Edge cases (общие)

- Голосовое < 2 сек → обрабатывается нормально
- Голосовое > 3 мин → предупреждение "обработка займёт больше времени"
- Фоновый шум → транскрипт с ошибками, бот обрабатывает как есть
- Транскрипт > 4000 символов → обрезается с уведомлением
- LLM недоступен → "Попробуй ещё раз" (без технических деталей)
- Транскрибация не удалась → "Не удалось распознать. Попробуй ещё раз"

## Критерии приёмки

- [ ] Бот принимает голосовое, показывает расшифровку + [📋 Извлечь задачи] [📝 Сделать резюме]
- [ ] 📋 → нумерованный список задач + rating 1-5
- [ ] 📝 → резюме от первого лица, без фидбека
- [ ] На "ааа, короче, надо позвонить маме" → "Позвонить маме" без мусора
- [ ] На "ну тот мужик, Аня знает" → задача "Уточнить у Ани"
- [ ] На "позвонить Ване, узнать номер у Пети" → "Узнать номер" стоит перед "Позвонить Ване"
- [ ] Несколько голосовых (буфер 3 сек) → единый контекст → один результат
- [ ] Если задач нет → явное сообщение
- [ ] Если задач > 10 → "разбей на несколько голосовых"
- [ ] Статус "Обрабатываю..." если > 5 сек
- [ ] Предупреждение при голосовом > 3 мин
- [ ] Транскрипт обрезается до 4000 символов с уведомлением
- [ ] Rating 1-5 после списка задач
- [ ] Rating < 5 → комментарий → согласие на сохранение аудио
- [ ] Аудио сохраняется только при согласии
- [ ] Пользователь регистрируется в БД при /start
- [ ] Триал: 20 → survey (4 вопроса, LLM sanity check, 2-strike) → +10 → блок
- [ ] /start — приветствие + инструкция + триал-инфо
- [ ] Не-голосовые → объяснение что делать
- [ ] Бот не падает при любом типе сообщения
- [ ] Время ответа < 30 сек для голосового до 1 мин
- [ ] Логи: user_id, duration, task_count; ошибки с timestamp

## Ограничения

- **Python/aiogram** — тот же стек, что МаркетБот (общий venv)
- **OpenRouter** — LLM провайдер (qwen/qwen-2.5-72b-instruct)
- **faster-whisper** — in-process, не HTTP-сервер (как в МаркетБоте)
- **SQLite** — users, voice_requests, feedback, survey_responses
- **Русский язык** — единственный в MVP
- **5 тестеров** — не нужна очередь, синхронный better-sqlite3 допустим
- **Trial: 20 + 10 = 30** — после 30 полная блокировка
- **Монетизация** — отложена на v2
- **Контекст между сессиями** — не делается в MVP (v2)
- **Напоминания** — не делаются в MVP (v2)
- **Экспорт в Notion/Todoist** — не делается (v3)

## Риски

| Риск | Митигация |
|------|-----------|
| LLM качество task extraction | LLM провайдер заменяемый через config |
| faster-whisper на CPU медленный | model: small, compute_type: int8, ~2x realtime |
| LLM токены исчерпаются | Мониторинг баланса, алерт при <$1 |
| Multi-voice race condition | Per-chat debounce buffer (3s) |
| better-sqlite3 блокирует event loop | Acceptable для 5 тестеров; async в v2 |

## Тестирование

- **Unit-тесты** — всегда, для каждой функции
- **Интеграционные** — полный pipeline: mock STT + mock LLM → результат
- **E2E** — ручной smoke-test с реальным ботом (5 сценариев)
